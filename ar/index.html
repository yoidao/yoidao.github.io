رموز أخطاء Youdao Zhiyun API / تشخيص واجهة الترجمة

هل أعادت واجهة الترجمة / القاموس / OCR / TTS من Youdao Zhiyun رمز errorCode؟ وضّح ثلاثة أمور أولًا: أهي مشكلة معامل/توقيع يمكن للعميل فحصها، أم مشكلة خادم/فوترة، هل ترتيب وترميز سلسلة التوقيع صحيحان، وهل تجاوزت QPS. أدخل رمز الخطأ أو كلمة مفتاحية (مثل 202 و411 و108 وتوقيع وحد وطابع زمني): أولًا المعنى الرسمي L1، ثم السبب الجذري والتشخيص L2، وأخيرًا مقارنة المنافسين وحالات واقعية. الوثائق الرسمية على دليل التكامل / رموز أخطاء Youdao Zhiyun.

الخطوة 1: صنّف حسب نطاق الرمز (L1 - أولًا خطأ مَن)

ينقسم errorCode في Youdao Zhiyun إلى أربعة نطاقات تقريبًا؛ حدّد قبل التجربة العمياء: ① 1xx معاملات (101 ناقص / 102 لغة / 103 طويل / 113 q فارغ / 116 strict) - يفحص العميل الإلزامي والقيم؛ ② 2xx مصادقة وتوقيع (108 معرّف / 202 توقيع / 203 IP / 205 منصة / 206 طابع زمني / 207 إعادة) - افحص سلسلة التوقيع وبيانات الاعتماد؛ ③ 3xx/5xx خادم (303 / 500 / 902000) - أعد المحاولة بـ request_id أو افتح تذكرة؛ ④ 4xx حساب/فوترة/حد (401 رصيد / 411 / 412) - افحص الرصيد وQPS. المزيد على توليد توقيع v3 وتشخيص 202.

معاملات101 · 102 · 103 · 113 · 116
فحص العميل: الحقول الإلزامية كاملة (q/from/to/appKey/sign/signType/curtime/salt)، رموز اللغة zh-CHS / en / auto، q غير فارغ/طويل، strict true/false.
مصادقة / توقيع108 · 111 · 202 · 203 · 205 · 206 · 207
SHA256 لـ appKey+input+salt+curtime+appSecret؛ curtime بتوقيت UTC بالثواني، salt من UUID لمنع الإعادة؛ قائمة IP البيضاء متوافقة مع نوع المنصة.
خدمة / مثيل110 · 310
كل تطبيق يربط مثيلًا لكل خدمة (الترجمة / TTS / OCR منفصلة)؛ تُفعّل ترجمة المجال في وحدة التحكم أولًا.
حساب / فوترة / حد401 · 411 · 412
لا تستنفد الرصيد / حزمة الأحرف (401 يوقف الخدمة)؛ تجاوز QPS (الأساس ~100) = 411: token bucket + تراجع أسي.
نتيجة / فك تشفير / أخرى201 · 303 · 500 · 902000
أخطاء الخادم: أعد بـ request_id أو تذكرة؛ في الواجهة المشفّرة افحص ترتيب DES / BASE64 / URLDecode.

الخطوة 2: أكثر فخّين شيوعًا → السبب الجذري (L2: التوقيع 202 والحد 411)

المفتاح: 90% من أخطاء Youdao API تتعطل عند «التوقيع 202» و«الحد 411». نادرًا ما يكون 202 مفتاحًا خاطئًا؛ غالبًا سلسلة توقيع v3 مبنية خطأ - sign = sha256(appKey + input + salt + curtime + appSecret)، حيث يأخذ input عند q>20 «أول 10 أحرف + طول q + آخر 10 أحرف» (حسب أحرف Unicode لا البايتات)، وq إذا ≤20. يُضاف ترميز UTF-8 والترتيب والمسافات. أما 411 فهو QPS فوق الباقة: قيّد المعدل + تراجع لدى العميل. أدناه لوحة الفحص الذاتي ثم المقارنة وتقديرات 2026.

تقدير 2026 من الوثائق الرسمية للمنصات (Youdao Zhiyun وBaidu وTencent Cloud وAlibaba Cloud وGoogle وDeepL): ليست قيمًا مضمونة أو مقيسة مباشرة، للاسترشاد فقط؛ تتغير الحصص وQPS وفق السياسة الرسمية:

أساس التقدير: مصادر مرجعية + استقراء زمني (وثائق 2026 الرسمية لـ Youdao / Baidu / Tencent / Alibaba / Google / DeepL)؛ يتغير وفق السياسة والإصدار؛ المرجع وحدة التحكم الرسمية الأحدث.

تشخيص سريع حسب سيناريو واقعي

  • يعمل محليًا، لكن كثير من 206 على السحابة/الحاوية: انحراف ساعة الحاوية يبطل curtime (التوقيع ~120s) → فعّل NTP، زامن الساعة مع المضيف؛ لا تخزّن curtime القديم.
  • الإنجليزية تُترجم جيدًا، والصينية الطويلة أحيانًا 202: قطع التقطيع المتعدد البايت حسب «البايت» → يجب أن يعدّ input أحرف Unicode (q>20: 10 + الطول + 10).
  • salt ثابت أو متزايد و207 في الاستدعاء الثاني: يجب أن يكون salt عشوائيًا (UUID) كل مرة وتحديث curtime؛ عند إعادة المحاولة وقّع من جديد.
  • ذروة التزامن في اختبار الحمل → كثير من 411: تجاوز QPS الأساس ~100 → token bucket + تراجع لدى العميل، أو اشترِ QPS من وحدة التحكم.
  • تطبيق جديد يستدعي النطق فيحصل على 110: مثيل الترجمة مرتبط، أما TTS فلا → اربط كل مثيل لكل خدمة في وحدة التحكم.
  • فجأة الكل 401: نفدت حزمة الأحرف / الرصيد → اشحن واضبط تنبيهات الرصيد؛ وفّر قناة احتياطية في المسارات الحرجة.
  • الفخ نفسه عند الانتقال من Baidu: q مُرمّز URL قبل التوقيع → يوقّع Youdao v3 بـ input (قاعدة القطع) لا بـ q المرمّز؛ وقّع أولًا ثم رمّز q عند الإرسال.
