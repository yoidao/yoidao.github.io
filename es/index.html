Códigos de error de la API de Youdao Zhiyun / diagnóstico de la API de traducción

¿La interfaz de traducción / diccionario / OCR / TTS de Youdao Zhiyun devolvió un errorCode? Aclara tres cosas primero: ¿es un problema de parámetro/firma que el cliente puede revisar, o del servidor/facturación?, ¿el orden y la codificación de la cadena de firma son correctos? y ¿se superó el QPS?. Escribe el código de error o palabra clave (p. ej. 202, 411, 108, firma, límite, timestamp): primero el significado oficial L1, luego la causa raíz y el diagnóstico L2, y por último la comparación con rivales y casos reales. La documentación oficial está en la guía de integración / códigos de error de Youdao Zhiyun.

Paso 1: clasifica por rango de código (L1, primero de quién es el fallo)

El errorCode de Youdao Zhiyun se divide en cuatro tramos; ubícalo por tramo antes de probar a ciegas: ① 1xx parámetros (101 falta / 102 idioma / 103 largo / 113 q vacío / 116 strict) — revisa obligatorios y valores en el cliente; ② 2xx auth y firma (108 ID / 202 firma / 203 IP / 205 plataforma / 206 timestamp / 207 repetición) — revisa la cadena de firma y credenciales; ③ 3xx/5xx servidor (303 / 500 / 902000) — reintenta con request_id o abre ticket; ④ 4xx cuenta/facturación/límite (401 saldo / 411 / 412) — revisa saldo y QPS. Más detalle en generación de firma v3 y diagnóstico del 202.

Paso 2: los dos fallos más frecuentes → causa raíz (L2: firma 202 y límite 411)

Clave: el 90% de los errores de la API de Youdao se atascan en «firma 202» y «límite 411». El 202 no suele ser una clave incorrecta, sino una cadena de firma v3 mal construida — sign = sha256(appKey + input + salt + curtime + appSecret), donde input con q>20 toma «primeros 10 caracteres + longitud de q + últimos 10 caracteres» (por caracteres Unicode, no bytes), y ≤20 toma q. Súmale codificación UTF-8, orden y espacios. El 411 es QPS por encima del plan: aplica limitación de tasa + backoff en el cliente. Diagnóstico rápido por escenario real

  • Funciona en local, pero al subir a la nube/contenedor aparecen muchos 206: deriva del reloj del contenedor que invalida curtime (firma válida ~120s) → activa NTP y sincroniza el reloj con el host; no caches curtime viejos.
  • El inglés traduce bien, pero el chino largo da 202 a veces: el truncado cortó el texto multibyte por «bytes» → input debe contar caracteres Unicode (q>20: 10 primeros + longitud + 10 últimos).
  • salt fijo o autoincremental y a la segunda llamada un 207: el salt debe ser aleatorio (UUID) cada vez y refrescar curtime; al reintentar, vuelve a firmar.
  • Picos de concurrencia en pruebas de carga → muchos 411: QPS base ~100 superado → token bucket + backoff en el cliente, o compra más QPS en la consola.
  • App nueva que llama a la pronunciación y da 110: la instancia de traducción está vinculada pero la de TTS no → vincula cada instancia por servicio en la consola.
  • De repente todo da 401: paquete de caracteres / saldo agotado → recarga y configura alertas de saldo; ten un canal de respaldo en rutas críticas.
  • El mismo fallo al migrar de Baidu: codificaste q con URL encode antes de firmar → Youdao v3 firma con input (regla de truncado), no con q codificado; firma primero y codifica q al enviar.
