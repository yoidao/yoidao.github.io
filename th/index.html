รหัสข้อผิดพลาด API Youdao Zhiyun / วินิจฉัย API แปลภาษา

อินเทอร์เฟซแปล / พจนานุกรม / OCR / TTS ของ Youdao Zhiyun คืนค่า errorCode? ก่อนอื่นทำสามอย่างให้ชัด: เป็นปัญหาพารามิเตอร์/ลายเซ็นที่ฝั่งไคลเอนต์ตรวจได้ หรือฝั่งเซิร์ฟเวอร์/การเรียกเก็บเงิน, ลำดับและการเข้ารหัสของสตริงลายเซ็นถูกต้องหรือไม่ และ QPS เกินหรือไม่ กรอก รหัสข้อผิดพลาดหรือคำสำคัญ (เช่น 202, 411, 108, ลายเซ็น, จำกัด, timestamp): ก่อนคือ ความหมายทางการ L1 จากนั้น ต้นเหตุและการวินิจฉัย L2 สุดท้ายคือการเปรียบเทียบคู่แข่งและกรณีจริง เอกสารทางการที่ คู่มือเชื่อมต่อ / รหัสข้อผิดพลาด Youdao Zhiyun

ขั้นที่ 1: แยกตามช่วงรหัส (L1 - ก่อนอื่นเป็นความผิดของใคร)

errorCode ของ Youdao Zhiyun แบ่งคร่าว ๆ เป็นสี่ช่วง; ระบุก่อนแทนการลองมั่ว: ① 1xx พารามิเตอร์ (101 ขาด / 102 ภาษา / 103 ยาว / 113 q ว่าง / 116 strict) - ไคลเอนต์ตรวจฟิลด์บังคับและค่า; ② 2xx ยืนยันตัวตนและลายเซ็น (108 ID / 202 ลายเซ็น / 203 IP / 205 แพลตฟอร์ม / 206 timestamp / 207 รีเพลย์) - ตรวจสตริงลายเซ็นและข้อมูลรับรอง; ③ 3xx/5xx เซิร์ฟเวอร์ (303 / 500 / 902000) - ลองใหม่ด้วย request_id หรือเปิด ticket; ④ 4xx บัญชี/เรียกเก็บเงิน/จำกัด (401 ยอดเงิน / 411 / 412) - ตรวจยอดเงินและ QPS เพิ่มเติมที่ การสร้างลายเซ็น v3 และวินิจฉัย 202

ขั้นที่ 2: สองกับดักที่พบบ่อยที่สุด → ต้นเหตุ (L2: ลายเซ็น 202 และจำกัด 411)

หัวใจ: 90% ของข้อผิดพลาด API Youdao ติดอยู่ที่ «ลายเซ็น 202» และ «จำกัด 411» 202 ไม่ค่อยเกิดจากคีย์ผิด แต่มักเป็น สตริงลายเซ็น v3 ที่ประกอบผิด - sign = sha256(appKey + input + salt + curtime + appSecret) โดย input เมื่อ q>20 ใช้ «10 อักขระแรก + ความยาว q + 10 อักขระท้าย» (ตามอักขระ Unicode ไม่ใช่ไบต์) และ q เมื่อ ≤20 บวกการเข้ารหัส UTF-8 ลำดับ และช่องว่าง ส่วน 411 คือ QPS เกินแพ็กเกจ: จำกัดอัตรา + backoff ที่ไคลเอนต์ ด้านล่างคือแผงตรวจสอบตนเอง ตามด้วยการเปรียบเทียบและประมาณการปี 2026

การจัดการข้อผิดพลาดและโควตาของ API แปลภาษา (ประมาณ 2026)

ประมาณการ 2026 จากเอกสารทางการของแต่ละแพลตฟอร์ม (Youdao Zhiyun, Baidu, Tencent Cloud, Alibaba Cloud, Google, DeepL): ไม่ใช่ค่าที่รับประกันหรือวัดเองโดยตรง ใช้อ้างอิงเท่านั้น; โควตาและ QPS เปลี่ยนตามนโยบายทางการ:

ฐานการประมาณ: แหล่งอ้างอิง + การคาดการณ์ตามเวลา (เอกสารทางการ 2026 ของ Youdao / Baidu / Tencent / Alibaba / Google / DeepL); เปลี่ยนตามนโยบายและเวอร์ชัน; ยึดคอนโซลทางการล่าสุด

วินิจฉัยรวดเร็วตามสถานการณ์จริง

  • ใช้ที่เครื่องได้ แต่ขึ้นคลาวด์/คอนเทนเนอร์แล้ว 206 จำนวนมาก: นาฬิกาคอนเทนเนอร์คลาดทำให้ curtime ใช้ไม่ได้ (ลายเซ็น ~120s) → เปิด NTP, ซิงก์นาฬิกากับโฮสต์; อย่าแคช curtime เก่า
  • แปลอังกฤษได้ดี แต่จีนยาวบางครั้ง 202: การตัดตัดมัลติไบต์ตาม «ไบต์» → input ต้องนับอักขระ Unicode (q>20: 10 + ความยาว + 10)
  • salt คงที่หรือเพิ่มทีละหนึ่ง แล้ว 207 ที่การเรียกครั้งที่สอง: salt ต้องสุ่ม (UUID) ทุกครั้งและรีเฟรช curtime; เมื่อ retry ให้เซ็นใหม่
  • พีคการทำงานพร้อมกันตอนทดสอบโหลด → 411 จำนวนมาก: QPS พื้นฐาน ~100 ถูกเกิน → token bucket + backoff ที่ไคลเอนต์ หรือซื้อ QPS เพิ่มในคอนโซล
  • แอปใหม่เรียกการออกเสียงแล้วได้ 110: instance แปลผูกแล้ว แต่ TTS ยังไม่ผูก → ผูกแต่ละ instance ต่อบริการในคอนโซล
  • จู่ ๆ ทั้งหมด 401: แพ็กอักขระ / ยอดเงินหมด → เติมเงินและตั้งการแจ้งเตือนยอดเงิน; เตรียมช่องทางสำรองในเส้นทางสำคัญ
  • กับดักเดียวกันเมื่อย้ายจาก Baidu: q ถูก URL-encode ก่อนเซ็น → Youdao v3 เซ็นด้วย input (กฎการตัด) ไม่ใช่ q ที่เข้ารหัส; เซ็นก่อน แล้วค่อย encode q ตอนส่ง
