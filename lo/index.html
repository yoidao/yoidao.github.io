ລະຫັດຂໍ້ຜິດພາດ Youdao Zhiyun API / ການວິນິດໄສ API ແປພາສາ

ອິນເຕີເຟດແປ / ວັດຈະນານຸກົມ / OCR / TTS ຂອງ Youdao Zhiyun ສົ່ງຄືນ errorCode ບໍ? ກ່ອນອື່ນ ໃຫ້ຊັດເຈນສາມຢ່າງ: ເປັນບັນຫາພາລາມິເຕີ/ລາຍເຊັນທີ່ client ກວດໄດ້ ຫຼື server/ການເກັບເງິນ, ລຳດັບ ແລະ ການເຂົ້າລະຫັດຂອງສະຕຣິງລາຍເຊັນຖືກຕ້ອງບໍ, ແລະ QPS ເກີນບໍ. ໃສ່ ລະຫັດຂໍ້ຜິດພາດ ຫຼື ຄຳສຳຄັນ (ເຊັ່ນ 202, 411, 108, ລາຍເຊັນ, ຈຳກັດ, timestamp): ກ່ອນແມ່ນ ຄວາມໝາຍທາງການ L1, ຈາກນັ້ນ ສາເຫດຫຼັກ ແລະ ການວິນິດໄສ L2, ສຸດທ້າຍ ການປຽບທຽບຄູ່ແຂ່ງ ແລະ ກໍລະນີຈິງ. ເອກະສານທາງການຢູ່ ຄູ່ມືເຊື່ອມຕໍ່ / ລະຫັດຂໍ້ຜິດພາດ Youdao Zhiyun.

ຂັ້ນຕອນ 1: ຈັດປະເພດຕາມຊ່ວງລະຫັດ (L1 - ກ່ອນອື່ນເປັນຄວາມຜິດຂອງໃຜ)

errorCode ຂອງ Youdao Zhiyun ແບ່ງປະມານສີ່ຊ່ວງ; ກຳນົດຕຳແໜ່ງກ່ອນລອງມົ່ວ: ① 1xx ພາລາມິເຕີ (101 ຂາດ / 102 ພາສາ / 103 ຍາວ / 113 q ຫວ່າງ / 116 strict) - client ກວດສິ່ງບັງຄັບ ແລະ ຄ່າ; ② 2xx ການຢືນຢັນ ແລະ ລາຍເຊັນ (108 ID / 202 ລາຍເຊັນ / 203 IP / 205 ແພລດຟອມ / 206 timestamp / 207 ຫຼິ້ນຄືນ) - ກວດສະຕຣິງລາຍເຊັນ ແລະ ຂໍ້ມູນຢືນຢັນ; ③ 3xx/5xx server (303 / 500 / 902000) - ລອງໃໝ່ດ້ວຍ request_id ຫຼື ຕັ້ງ ticket; ④ 4xx ບັນຊີ/ການເກັບເງິນ/ຈຳກັດ (401 ຍອດເງິນ / 411 / 412) - ກວດຍອດເງິນ ແລະ QPS. ເພີ່ມເຕີມຢູ່ ການສ້າງລາຍເຊັນ v3 ແລະ ການວິນິດໄສ 202.

ຂັ້ນຕອນ 2: ສອງກັບດັກທີ່ພົບເລື້ອຍທີ່ສຸດ → ສາເຫດຫຼັກ (L2: ລາຍເຊັນ 202 ແລະ ຈຳກັດ 411)

ຫົວໃຈ: 90% ຂອງຂໍ້ຜິດພາດ Youdao API ຕິດຢູ່ທີ່ «ລາຍເຊັນ 202» ແລະ «ຈຳກັດ 411». 202 ບໍ່ຄ່ອຍແມ່ນກະແຈຜິດ; ສ່ວນຫຼາຍແມ່ນ ສະຕຣິງລາຍເຊັນ v3 ທີ່ປະກອບຜິດ - sign = sha256(appKey + input + salt + curtime + appSecret), ເຊິ່ງເມື່ອ q>20 input ເອົາ «10 ຕົວອັກສອນທຳອິດ + ຄວາມຍາວ q + 10 ຕົວອັກສອນສຸດທ້າຍ» (ຕາມຕົວອັກສອນ Unicode ບໍ່ແມ່ນໄບຕ໌), ແລະ q ເມື່ອ ≤20. ບວກການເຂົ້າລະຫັດ UTF-8, ລຳດັບ ແລະ ຊ່ອງຫວ່າງ. 411 ແມ່ນ QPS ເກີນແພັກເກດ: ຈຳກັດອັດຕາ + backoff ທີ່ client. ດ້ານລຸ່ມແມ່ນແຜງກວດສອບຕົນເອງ, ຈາກນັ້ນການປຽບທຽບ ແລະ ການຄາດຄະເນ 2026.

ປະມານ 2026 ຈາກເອກະສານທາງການຂອງແພລດຟອມ (Youdao Zhiyun, Baidu, Tencent Cloud, Alibaba Cloud, Google, DeepL): ບໍ່ແມ່ນຄ່າທີ່ສັນຍາ ຫຼື ວັດແທກໂດຍກົງ, ສຳລັບອ້າງອີງເທົ່ານັ້ນ; ໂຄຕ້າ ແລະ QPS ປ່ຽນຕາມນະໂຍບາຍທາງການ:

ພື້ນຖານປະມານ: ແຫຼ່ງອ້າງອີງ + ການຄາດຄະເນຕາມເວລາ (ເອກະສານທາງການ 2026 ຂອງ Youdao / Baidu / Tencent / Alibaba / Google / DeepL); ປ່ຽນຕາມນະໂຍບາຍ ແລະ ເວີຊັນ; ຄອນໂຊລທາງການລ່າສຸດເປັນຫຼັກ.

ການວິນິດໄສໄວຕາມສະຖານະການຈິງ

  • ເຮັດວຽກໃນ local ໄດ້, ແຕ່ຢູ່ cloud/container ມີ 206 ຫຼາຍ: ໂມງ container ເລື່ອນເຮັດໃຫ້ curtime ໃຊ້ບໍ່ໄດ້ (ລາຍເຊັນ ~120s) → ເປີດ NTP, ຊິງໂມງກັບ host; ຢ່າແຄຊ curtime ເກົ່າ.
  • ອັງກິດແປດີ, ຈີນຍາວບາງເທື່ອ 202: truncation ຕັດ multibyte ຕາມ «ໄບຕ໌» → input ຕ້ອງນັບຕົວອັກສອນ Unicode (q>20: 10 + ຄວາມຍາວ + 10).
  • salt ຄົງທີ່ ຫຼື ເພີ່ມຂຶ້ນ ແລະ 207 ໃນການເອີ້ນຄັ້ງສອງ: salt ຕ້ອງສຸ່ມ (UUID) ທຸກຄັ້ງ ແລະ ໃໝ່ curtime; ເມື່ອ retry ໃຫ້ເຊັນໃໝ່.
  • ຈຸດສູງສຸດການເຮັດວຽກພ້ອມກັນໃນການທົດສອບໂຫຼດ → 411 ຫຼາຍ: QPS ພື້ນຖານ ~100 ເກີນ → token bucket + backoff ທີ່ client, ຫຼື ຊື້ QPS ໃນຄອນໂຊລ.
  • ແອັບໃໝ່ເອີ້ນການອອກສຽງແລ້ວໄດ້ 110: instance ແປໄດ້ຜູກ, TTS ບໍ່ → ຜູກແຕ່ລະ instance ຕໍ່ບໍລິການໃນຄອນໂຊລ.
  • ທັນໃດທັງໝົດ 401: ແພັກຕົວອັກສອນ / ຍອດເງິນໝົດ → ເຕີມ ແລະ ຕັ້ງການເຕືອນຍອດເງິນ; ມີຊ່ອງສຳຮອງໃນເສັ້ນທາງສຳຄັນ.
  • ກັບດັກດຽວກັນເມື່ອຍ້າຍຈາກ Baidu: q ຖືກ URL-encode ກ່ອນເຊັນ → Youdao v3 ເຊັນດ້ວຍ input (ກົດການຕັດ), ບໍ່ແມ່ນ q ທີ່ເຂົ້າລະຫັດ; ເຊັນກ່ອນ, ເຂົ້າລະຫັດ q ເມື່ອສົ່ງ.
