유다오 즈윈 API 오류 코드 / 번역 API 문제 해결 조회

유다오 즈윈 번역 / 사전 / OCR / TTS 인터페이스가 errorCode 를 반환했나요? 먼저 세 가지를 정리하세요 — 클라이언트가 자체 점검 가능한 파라미터/서명 문제인지, 서버/과금 문제인지, 서명 문자열의 순서와 인코딩이 맞는지, QPS 를 초과했는지. 받은 오류 코드나 키워드(예: 202, 411, 108, 서명, 레이트리밋, 타임스탬프)를 입력하면 먼저 L1 공식 의미, 그다음 L2 근본 원인과 점검, 마지막으로 경쟁사 비교와 실제 사례를 확인할 수 있습니다. 공식 연동 문서는 유다오 즈윈 API 오류 코드 / 연동 가이드 에서.

1단계: 코드 구간으로 분류 (L1·누구의 문제인지부터)

유다오 즈윈의 errorCode 는 대략 네 구간으로 나뉩니다. 먼저 구간으로 위치를 잡으세요: ① 1xx 파라미터류(101 누락 / 102 언어 / 103 길이초과 / 113 q 빈값 / 116 strict) — 클라이언트에서 필수값과 형식을 점검; ② 2xx 인증·서명류(108 앱ID / 202 서명 / 203 IP / 205 플랫폼 / 206 타임스탬프 / 207 재전송) — 서명 문자열과 자격증명 확인; ③ 3xx/5xx 서버류(303 / 500 / 902000) — request_id 로 재시도하거나 티켓; ④ 4xx 계정/과금/레이트리밋(401 잔액 / 411 / 412) — 잔액과 QPS 확인. 더 자세한 비교는 유다오 API 서명(v3) 생성과 202 점검.

파라미터류101 · 102 · 103 · 113 · 116
클라이언트 자체 점검: 필수 필드(q/from/to/appKey/sign/signType/curtime/salt) 누락 없음, 언어 코드는 zh-CHS / en / auto, q 는 비어있지 않고 길이 초과 없음, strict 는 true/false.
인증 / 서명류108 · 111 · 202 · 203 · 205 · 206 · 207
서명 문자열 appKey+input+salt+curtime+appSecret 를 SHA256; curtime 은 초 단위 UTC, salt 는 재전송 방지를 위해 UUID; IP 화이트리스트와 앱 플랫폼 유형 정렬.
서비스 / 인스턴스류110 · 310
앱은 서비스마다 인스턴스를 따로 바인딩(번역 / 음성합성 TTS / OCR 각각 독립); 도메인 번역은 콘솔에서 먼저 활성화.
계정 / 과금 / 레이트리밋류401 · 411 · 412
잔액 / 문자 리소스 팩 소진 주의(401 잔액부족 정지); QPS(기본 약 100) 초과 시 411, 클라이언트는 토큰 버킷 레이트 리밋 + 지수 백오프.
결과 / 복호화 / 기타류201 · 303 · 500 · 902000
서버류 예외는 request_id 로 재시도하거나 티켓; 암호화 인터페이스는 DES / BASE64 / URLDecode 순서 확인.

2단계: 최빈 함정 둘 → 근본 원인 (L2·서명 202 와 레이트리밋 411)

핵심: 유다오 API 오류의 90% 는 「서명 202」와 「레이트리밋 411」에서 막힙니다. 202 는 키가 틀린 게 아니라 대개 v3 서명 문자열을 잘못 만든 것입니다 — sign = sha256(앱ID + input + salt + curtime + 앱시크릿), 여기서 input 은 q 길이>20 일 때 「앞 10자 + q 길이 + 뒤 10자」, ≤20 이면 q 자체(바이트가 아닌 유니코드 문자 기준). 거기에 UTF-8 인코딩·순서·앞뒤 공백 함정이 겹칩니다. 411 은 QPS 가 요금제 상한을 넘은 것이므로 클라이언트 레이트 리밋 + 백오프가 필요합니다. 아래는 호출 전 자체 점검 패널 예시이고, 그 아래는 경쟁사 비교와 2026 추정입니다.

실제 사례 빠른 판별

  • 로컬은 되는데 클라우드/컨테이너로 올리면 206 대량 발생: 컨테이너 시계 드리프트로 curtime 무효(서명 유효기간 약 120s) → NTP 동기화, 컨테이너가 호스트 시계와 동기, 오래된 curtime 캐시 금지.
  • 영어 번역은 정상인데 긴 중국어 번역에서 가끔 202: truncate 가 멀티바이트 중국어를 「바이트」로 잘랐기 때문 → input 은 유니코드 문자 수 기준(q>20 이면 앞 10자 + q 길이 + 뒤 10자).
  • salt 를 고정/증가값으로 쓰고 두 번째 호출에서 207: salt 는 매번 랜덤(UUID)으로 만들고 curtime 갱신, 재시도 시 그대로 재전송 말고 다시 서명.
  • 부하 테스트 순간 동시성에서 대량 411: 기본 QPS 약 100 초과로 제한 → 클라이언트 토큰 버킷 레이트 리밋 + 지수 백오프, 또는 콘솔에서 QPS 추가 구매.
  • 새 앱으로 바로 번역 발음 호출 시 110: 번역 인스턴스는 바인딩됐지만 음성합성(TTS) 인스턴스 미바인딩 → 콘솔에서 서비스별로 인스턴스 바인딩.
  • 갑자기 전부 401: 문자 리소스 팩 / 잔액 소진으로 정지 → 충전·리소스 팩 구매 및 잔액 알림 설정, 핵심 경로엔 대체 채널.
  • 바이두에서 옮겨온 동일 함정: 서명 결합 전에 q 를 URL 인코딩함 → 유다오 v3 는 input(절단 규칙)으로 서명, q 를 먼저 인코딩하지 않음; 서명을 먼저 하고 요청 시점에 q 를 URL 인코딩.
